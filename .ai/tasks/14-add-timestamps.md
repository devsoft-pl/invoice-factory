# Task 14: Add created_at / updated_at where appropriate

## Priority: LOW
## Status: todo
## Estimated scope: small

## Problem

Some models lack audit timestamps, which makes debugging and auditing difficult. Not every model needs them — the decision should be deliberate.

## Current state

**Already have dates — do NOT touch:**
- `Invoice` — `create_date`, `sale_date`, `settlement_date`, `payment_date`
- `ExchangeRate` — `date`

**Potentially useful:**
- `Company` — useful to know when a company was added
- `Person` — useful to know when a contact was added
- `User` — `AbstractBaseUser` provides `last_login`; `date_joined` is missing

**Likely not needed:**
- `Item` — always linked to an Invoice which already has dates
- `VatRate`, `Country`, `Currency` — simple reference data, rarely changed
- `Year` — auto-generated by signals
- `Accountant`, `SummaryRecipient` — configuration, not transactional data

## Files to modify (if timestamps are added)

- `base/mixins.py` — abstract `TimestampedModel`
- `companies/models.py`
- `persons/models.py`
- `users/models.py` — add `date_joined` directly (User inherits from AbstractBaseUser)

## Steps

1. Confirm with the user which models should receive timestamps — do not add blindly
2. Create `TimestampedModel` in `base/mixins.py`
3. Add inheritance only to the selected models
4. For User: add `date_joined = DateTimeField(auto_now_add=True)` directly on the model
5. Run `make makemigrations` — migration will require a default for existing rows (`django.utils.timezone.now`)
6. Run `make migrate` and `pytest .`

## Acceptance criteria

- Timestamps added ONLY where they are genuinely useful
- `Invoice` and `ExchangeRate` remain unchanged (they already have dates)
- Migrations generated and applied
- Tests pass
